## Project Overview

PhishGuard classifies email messages as **Phishing** or **Legitimate** and provides:

| Output       | Description                                             |
| ------------ | ------------------------------------------------------- |
| Label        | `Phishing` or `Legitimate`                              |
| Probability  | 0.00 вҖ“ 1.00 phishing confidence score                   |
| Risk level   | `Low` (< 0.33) В· `Medium` (0.33вҖ“0.66) В· `High` (вүҘ 0.66) |
| Explanations | Top 3 contributing features (metadata + TF-IDF signals) |

## Quick Start (Docker Compose)

> вҡ пёҸ You must train the model before running Docker Compose, because the
> pre-trained `stacking_pipeline.joblib` file is not committed to the repo.

### Step 1 вҖ“ Train the model

```bash
# (From the repo root)
# Create and activate a virtual environment
python -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate

# Install backend dependencies
pip install -r backend/requirements.txt

# Train on full datasets (reads from /mnt/data/)
python backend/train.py

# OR for a fast development run (~2 000 rows per dataset):
python backend/train.py --sample 2000
```

Training output: `backend/models/stacking_pipeline.joblib`  
Expected runtime: 5вҖ“20 minutes (full) В· 1вҖ“3 minutes (--sample 2000)

### Step 2 вҖ“ Spin up both services

```bash
docker compose up --build
```

| Service      | URL                          |
| ------------ | ---------------------------- |
| Frontend     | http://localhost:3000        |
| Backend API  | http://localhost:8000        |
| OpenAPI Docs | http://localhost:8000/docs   |
| Health check | http://localhost:8000/health |

Classify an email.

**Request body:**

## Deployment

### Netlify / Vercel (Frontend only)

```bash
cd frontend
npm run build      # produces ./dist/
```

Upload `frontend/dist/` to Netlify/Vercel. Set environment variable:

- `VITE_API_URL` вҶ’ your deployed backend URL (e.g. `https://api.phishguard.example.com`)

### Docker production deployment

```bash
# Build backend
docker build -t phishguard-backend ./backend

# Build frontend (inject API URL at build time)
docker build \
  --build-arg VITE_API_URL=https://api.phishguard.example.com \
  -t phishguard-frontend ./frontend

# Run
docker run -d -p 8000:8000 -v ./backend/models:/app/models phishguard-backend
docker run -d -p 3000:80 phishguard-frontend
```

### Rate limiting (production)

Add nginx or Caddy in front of the backend with rate-limit rules, e.g.:

```nginx
# nginx upstream config
limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;

location /predict {
    limit_req zone=api burst=20 nodelay;
    proxy_pass http://backend:8000;
}
```

## ML Pipeline Details

### Datasets

| File                  | Description              | Label column            |
| --------------------- | ------------------------ | ----------------------- |
| `spam (1).csv`        | Spam/ham email corpus    | `label` (`spam`/`ham`)  |
| `phishing_email.csv`  | Phishing email dataset   | `label` or `Email Type` |
| `TREC_05_cleaned.csv` | TREC 2005 spam benchmark | `label` (`spam`/`ham`)  |

All labels are mapped to binary `is_phish` (1 = phishing/spam, 0 = legitimate).

### Feature Engineering (14 metadata features)

| Feature                  | Description                                |
| ------------------------ | ------------------------------------------ |
| `has_url`                | Email contains at least one URL            |
| `num_urls`               | Count of URLs                              |
| `has_ip_in_url`          | URL contains a raw IP address              |
| `has_html_tags`          | Body contains HTML markup                  |
| `num_exclamations`       | Count of `!` characters                    |
| `num_question_marks`     | Count of `?` characters                    |
| `num_dollars`            | Count of `$` characters                    |
| `num_digits`             | Count of digit characters                  |
| `num_suspicious_words`   | Count of phishing-indicator words          |
| `has_suspicious_words`   | Binary: at least one suspicious word       |
| `sender_domain_mismatch` | From: domain вү  link domains (if available) |
| `length_subject`         | Character length of subject                |
| `length_body`            | Character length of body                   |
| `upper_ratio`            | Fraction of uppercase letters              |

### Risk Level Thresholds

| Phishing probability | Risk level |
| -------------------- | ---------- |
| p < 0.33             | рҹҹў Low     |
| 0.33 вүӨ p < 0.66      | рҹҹЎ Medium  |
| p вүҘ 0.66             | рҹ”ҙ High    |

### Cross-Validation Strategy

- `StratifiedKFold(n_splits=5)` inside `StackingClassifier`
- Base learners trained on 4/5 folds вҶ’ produce out-of-fold probabilities
- Meta-learner (`LogisticRegression`) trained on the OOF probabilities
- Final evaluation on a completely held-out test set (20% of data)

## Security & Privacy Notes

- **Email bodies are NOT logged** in the API server. Only metadata (subject length, body length, IP) is logged.
- In production, place the backend behind a reverse proxy (nginx/Caddy) with rate limiting (e.g. 10 req/s per IP).
- The Docker images run as non-root users (`appuser` in backend, `nginx` in frontend).
- Set `CORS_ORIGINS` environment variable in production to restrict allowed origins.
